# Spotify Orchestrator Service — Design Spec

## Overview

Build a NestJS orchestrator service (`/apps/orchestrator`) that takes an LLM-generated song list, resolves each song to a Spotify track, creates a playlist on the user's Spotify account, and adds the resolved tracks.

**Scope:** NestJS service only. The data-engine call is mocked for now. Spotify OAuth is handled externally (the web app passes an access token).

---

## Input: LLM Song List Schema

The data-engine's `POST /recommend` endpoint returns (mocked for now):

```json
[
  {
    "title": "Song Name",
    "artist": "Artist Name",
    "source": "user_library" | "new_suggestion"
  }
]
```

- `source: "user_library"` — song exists in the user's library (still needs Spotify URI resolution)
- `source: "new_suggestion"` — LLM-suggested song not in the user's library

---

## API Contract

### `POST /playlists/generate`

**Request:**
```json
{
  "eventDescription": "Late night study session",
  "playlistName": "Study Vibes",        // optional, auto-generated from event if missing
  "isPublic": false,                     // optional, defaults to false
  "spotifyAccessToken": "Bearer ..."     // user's Spotify access token
}
```

**Response (success):**
```json
{
  "playlistId": "spotify_playlist_id",
  "playlistUrl": "https://open.spotify.com/playlist/...",
  "tracksAdded": 18,
  "tracksNotFound": ["Song X by Artist Y", "Song Z by Artist W"],
  "totalRequested": 20
}
```

**Response (error):**
```json
{
  "error": "SPOTIFY_AUTH_EXPIRED" | "NO_TRACKS_RESOLVED" | "PLAYLIST_CREATION_FAILED",
  "message": "Human-readable description"
}
```

---

## Data Flow

```
Web App
  |
  |  POST /playlists/generate
  |  { eventDescription, spotifyAccessToken, playlistName?, isPublic? }
  v
NestJS Orchestrator
  |
  |  1. Call data-engine POST /recommend (mocked for now)
  |     Input: { eventDescription }
  |     Output: [{ title, artist, source }]
  |
  |  2. For each song: Spotify Search API
  |     q = "track:{title} artist:{artist}"
  |     → take first result → collect track URI
  |     → skip + log warning if no match
  |
  |  3. Create Playlist via Spotify API
  |     POST /users/{user_id}/playlists
  |     { name, public, description }
  |
  |  4. Add tracks to playlist
  |     POST /playlists/{id}/tracks
  |     { uris: ["spotify:track:..."] }
  |
  v
Return { playlistId, playlistUrl, tracksAdded, tracksNotFound }
```

## Key Design Decisions

### Spotify API Library
Use `spotify-web-api-node` — mature, well-maintained, covers all needed endpoints.

### Auth Strategy
The orchestrator does NOT handle OAuth. The web app performs Spotify login and passes the access token in the request body. The orchestrator sets this token on the Spotify client per-request.

### Track Resolution
Search query format: `track:{title} artist:{artist}`. Take the first result. If no result, skip the track and include it in `tracksNotFound`.

All songs are resolved (both `user_library` and `new_suggestion`) since we need Spotify URIs for all of them.

### Playlist Creation
- Name: provided by caller, or auto-generated as `"JamOn: {eventDescription}"` (truncated to 100 chars)
- Visibility: configurable via `isPublic` parameter, defaults to `false`
- Description: `"Generated by JamOn for: {eventDescription}"`

### Data-Engine Integration (Mocked)
A `DataEngineService` with a `getRecommendations(eventDescription: string)` method. Initially returns a hardcoded song list. Will be replaced with an HTTP call to the Python service's `POST /recommend` when the FastAPI wrapper is built.

## Error Handling

| Scenario | Behavior |
|---|---|
| Invalid/expired Spotify token | Return `{ error: "SPOTIFY_AUTH_EXPIRED" }` with 401 status |
| All songs fail Spotify search | Return `{ error: "NO_TRACKS_RESOLVED" }` — don't create empty playlist |
| Some songs not found | Skip them, add to `tracksNotFound`, create playlist with rest |
| Playlist creation fails | Return `{ error: "PLAYLIST_CREATION_FAILED" }` with details |
| Data-engine returns empty | Return `{ error: "NO_TRACKS_RESOLVED" }` |

## Testing Strategy

- **Unit tests** for `SpotifyService` — mock `spotify-web-api-node` calls, test search/create/add logic, test error paths (no results, expired token, etc.)
- **Integration test** for the controller endpoint — mock the Spotify service, verify the full request→response flow
- **Manual E2E** — call the endpoint with a real Spotify token, verify playlist appears in the user's account
